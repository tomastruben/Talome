This is not a chatbot. Talome's AI assistant is an autonomous agent with 219 purpose-built tools that plans multi-step operations, selects the right tools for each step, executes against your real infrastructure, and reasons about failures to recover. When you say "set up a media stack," it doesn't hand you a checklist — it installs five apps, wires their APIs together, syncs indexers, verifies health, and reports back with URLs.

The difference between a chatbot and an agent: a chatbot answers questions. An agent does work.

## Architecture

Every message you send passes through a precise pipeline before a single tool executes.

<Mermaid chart={`graph TD
    A[User sends message] --> B[Build system prompt]
    B --> C[Inject memories - top 10]
    C --> D[Add page context + stack status]
    D --> E[Select tools via keyword routing]
    E --> F[Security gateway wraps tools]
    F --> G[Stream to AI model]
    G --> H{Tool calls?}
    H -->|Yes| I[Execute tools - max 10]
    I --> J[Audit log entry]
    J --> G
    H -->|No| K[Stream text response]
`} />

### 1. System Prompt (Cached)

The system prompt is approximately 5,000 tokens and is cached using Anthropic's prompt caching. It contains the AI's identity, every behavioral rule, response formatting patterns, app-specific instructions (media volume conventions, Audiobookshelf library setup flow, arr wiring sequences), and the zero-config execution policy. Because it's cached, the first tokens of every conversation cost near zero — you only pay for new messages and tool results.

### 2. Dynamic Context Injection

Before the AI sees your message, Talome injects live context into the conversation:

- **Memories** — the top 10 most relevant memories about you, ranked by a composite score of recency, access frequency, and confidence
- **Page context** — which dashboard page you're on (the AI knows if you're looking at containers, media, or settings)
- **Stack status** — which feature stacks are installed and their current health
- **Configured apps** — which integrations have URLs and API keys set, so the AI knows what tools are available

### 3. Tool Selection (Keyword Routing + Domain Filtering)

Not all 219 tools load for every message. The tool registry uses a two-layer filter:

**Layer 1: Domain activation.** Each tool belongs to a domain. A domain's tools only load when the corresponding app is configured in Settings. If you haven't set up Jellyfin, the 6 Jellyfin tools don't exist in the AI's context.

**Layer 2: Message keyword routing.** Even among active domains, tools are further filtered by scanning your message for domain-relevant keywords. If your message mentions "torrent" or "download speed," the qBittorrent domain loads. If it mentions "indexer" or "quality profile," the arr domain loads. If no optional domain matches (ambiguous message), all active tools load as a fallback.

This keeps the per-message tool count low — typically 80-120 tools instead of 219 — which directly improves the AI's tool selection accuracy.

### 4. Security Gateway

Every tool execution passes through the security gateway, which enforces one of three modes:

| Mode | Read Tools | Modify Tools | Destructive Tools |
|------|-----------|-------------|------------------|
| **Permissive** | Execute freely | Execute freely | Execute freely |
| **Cautious** (default) | Execute freely | Execute freely | Require `confirmed: true` — the AI asks you first |
| **Locked** | Execute freely | Blocked | Blocked |

You can change the security mode in **Settings > Security**. The default "cautious" mode means the AI can read anything and modify anything, but destructive operations like uninstalling apps, deleting containers, or pruning Docker resources require your explicit confirmation.

### 5. Streaming Execution

The AI streams responses using the Vercel AI SDK's `streamText()` with a maximum of 10 tool calls per turn (`stepCountIs(10)`). Each tool call appears as a real-time execution card in the chat UI showing the tool name, parameters, and result. The AI processes tool results and continues reasoning — it can chain multiple tool calls in a single response to complete multi-step tasks.

## Tool Tiers

Every tool has an assigned security tier that determines how the security gateway handles it:

### Read Tier
Always execute, in any security mode. These tools only observe — they never change state.

Examples: `list_containers`, `get_system_stats`, `search_apps`, `recall`, `arr_get_status`, `jellyfin_list_libraries`, `get_disk_usage`, `diagnose_app`

### Modify Tier
Execute freely in permissive and cautious modes. Blocked in locked mode. These tools change state but are generally reversible.

Examples: `install_app`, `start_container`, `restart_app`, `wire_apps`, `set_app_env`, `remember`, `arr_add_download_client`, `create_automation`

### Destructive Tier
Execute freely in permissive mode. Require explicit confirmation in cautious mode (the AI asks before proceeding). Blocked in locked mode. These operations are difficult or impossible to reverse.

Examples: `uninstall_app`, `prune_resources`, `cleanup_docker`, `delete_group`, `apply_change`, `remove_network`, `delete_file`

## Memory System

The AI remembers what you tell it — permanently. Memories persist in SQLite across conversations, sessions, and restarts.

### Memory Types

| Type | Purpose | Example |
|------|---------|---------|
| **Preference** | How you like things configured | "I prefer Jellyfin over Plex" |
| **Fact** | Concrete details about your setup | "Media drive is mounted at /mnt/nas" |
| **Context** | Situational information | "Setting up a home lab for the first time" |
| **Correction** | Things the AI got wrong that it shouldn't repeat | "Port 8080 is taken by Apache" |

### How Ranking Works

Each conversation turn, the top 10 memories are injected into the AI's context. The ranking algorithm scores every memory on three factors:

1. **Recency** — more recently created or accessed memories score higher
2. **Access frequency** (`accessCount`) — memories the AI references often score higher
3. **Confidence** — memories with higher confidence scores (set at creation, boosted by confirmations) score higher

When the AI stores a new memory, it checks existing memories for bigram similarity. If a new memory is more than 80% similar to an existing one, the duplicate is rejected — preventing the same fact from being stored repeatedly across conversations.

<Accordions>
<Accordion title="Source">
`apps/core/src/db/memories.ts` — `getTopMemories()` (line 57) implements the composite ranking: 40% recency + 30% access frequency + 30% confidence, computed in a single SQL `ORDER BY` expression (lines 67-71). Deduplication uses a Dice coefficient on bigrams in the `similarity()` function (line 8), with the 80% threshold checked in `writeMemory()` at line 45.
</Accordion>
</Accordions>

### Memory Tools

| Tool | What It Does |
|------|-------------|
| `remember` | Store a new memory with type and confidence |
| `recall` | Search memories by keyword (semantic-ish, bigram matching) |
| `forget` | Delete a specific memory by ID |
| `update_memory` | Modify an existing memory's content or confidence |
| `list_memories` | Show all stored memories with metadata |

### Memory in Practice

Memories are stored implicitly when you share information, and explicitly when you ask:

```
You: My media is stored at /Volumes/Media Vault/Media

AI: Got it. I'll remember that your media is at /Volumes/Media Vault/Media.
    [remember: type=fact, content="Media files stored at /Volumes/Media Vault/Media"]

--- later, in a different conversation ---

You: Install Jellyfin

AI: [recall: "media"]
    Installing Jellyfin with your media directory at /Volumes/Media Vault/Media.
    [install_app: jellyfin, volumeMounts: { media: "/Volumes/Media Vault/Media" }]
```

The AI didn't ask where your media was the second time. It remembered.

## The Zero-Config Philosophy

Traditional self-hosted apps require you to install an app, then open its web UI, navigate to settings, find the right field, paste a URL or API key, save, restart, and repeat for every connected service. Talome's AI eliminates this entirely.

The system prompt contains a strict **Config-First Execution Policy**:

1. Use dedicated configuration tools first (`overseerr_configure_sonarr`, `arr_add_download_client`, etc.)
2. If a dedicated tool fails, fall back to config-file automation (`read_app_config_file` / `write_app_config_file`)
3. Only ask the user for values that are truly unavailable (API keys they haven't provided)
4. Never output "manual configuration required" if a tool or config-file path exists

This means the AI will never tell you to "go to Sonarr's settings page and add qBittorrent as a download client." It will call `arr_add_download_client` and do it.

## Multi-Provider Support

Talome supports three AI providers:

| Provider | Default Model | Best For |
|----------|--------------|----------|
| **Anthropic** | Claude Haiku 4.5 | Best tool use, recommended default |
| **OpenAI** | GPT-4o | Alternative cloud option |
| **Ollama** | Local models | Fully offline, privacy-first |

Configure your provider in **Settings > AI Provider**. The system prompt and tool architecture work identically across all providers — only the model's reasoning quality varies.

## What the AI Can Create

Beyond managing existing infrastructure, the AI assistant can create new resources through conversation:

- **Custom tools** -- define new tools at runtime via `create_tool`. Describe a capability (e.g., "check my UPS battery status via NUT") and the AI generates a working tool with Zod schema, execution function, and keyword triggers.
- **Dashboard widgets** -- create and configure widgets programmatically via `create_widget_manifest`. Describe what metrics or data you want to see and the AI builds a widget that appears on your dashboard.
- **Automations** -- describe a workflow in plain English and the AI generates the full automation with triggers, conditions, and steps via `create_automation`.
- **App blueprints** -- describe an app you want to build and the AI generates a complete Docker Compose stack with manifest, health checks, and optionally a full web UI via `design_app_blueprint`.
- **Integration wiring** -- the AI can configure cross-app connections (download clients, indexer sync, media server linking) through tools like `wire_apps`, `overseerr_configure_sonarr`, and `arr_add_download_client`.
- **Codebase changes** -- the AI can modify Talome's own TypeScript source code via the self-improvement pipeline (`plan_change` / `apply_change`), with automatic TypeScript validation and rollback.

Everything is conversational. You describe the intent, the AI builds the implementation.

## Custom System Prompts

You can append custom instructions to the system prompt in **Settings > AI**. Common uses:

- Set a preferred language: "Always respond in Spanish"
- Define naming conventions: "Use kebab-case for all container names"
- Add domain knowledge: "My NAS is a Synology DS920+ at 192.168.1.100"

Custom prompts are appended after the default system prompt and do not override it.

## Limitations

- **No internet access** — the AI works with your local system. It cannot browse the web, access external APIs, or download files from the internet (unless you've configured a web search tool).
- **Destructive operations need confirmation** — in the default "cautious" mode, operations like uninstalling apps, deleting files, or pruning Docker resources require you to confirm.
- **10 tool calls per turn** — the AI can execute up to 10 tool calls in a single response. For operations requiring more steps, it completes across multiple turns.
- **Zod validation on all inputs** — every tool parameter is validated with a Zod schema. The AI cannot pass malformed data to any tool.
- **No `eval()`** — the AI never executes arbitrary code strings. All operations go through typed, validated tool definitions.
- **Shell commands are restricted** — `run_shell` only executes commands you explicitly request and logs every execution to the audit trail.
